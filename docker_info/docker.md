# Docker Guide & Reference

Docker is an open-source platform that automates the deployment, scaling, and management of applications inside lightweight, portable containers.

---

## 1. Core Concepts

*   **Container vs. Virtual Machine (VM):**
    *   **VMs:** Include a full guest OS, hypervisor, and virtualized hardware. They are heavy (GBs) and slow to start.
    *   **Containers:** Share the host OS kernel and isolate the application processes. They are extremely lightweight (MBs) and start instantly.
*   **Image:** A read-only template with instructions for creating a Docker container. It includes the application code, runtime, libraries, environment variables, and config files.
*   **Container:** A runnable instance of an image. You can start, stop, move, or delete a container using the Docker API or CLI.
*   **Registry:** A registry stores Docker images (e.g., Docker Hub, AWS ECR, GitHub Packages).
*   **Volume:** A mechanism for persisting data generated by and used by Docker containers outside the container's lifecycle.
*   **Network:** Allows containers to communicate with each other and the host system.

## 2. Essential Docker Commands

### Image Operations
*   `docker build -t <image_name>:<tag> .` — Build an image from a Dockerfile in the current directory.
*   `docker images` — List all locally stored Docker images.
*   `docker rmi <image_id_or_name>` — Remove one or more images.
*   `docker pull <image_name>` — Pull an image from a registry.
*   `docker push <image_name>` — Push an image to a registry.

### Container Operations
*   `docker run -d -p <host_port>:<container_port> --name <container_name> <image_name>` — Run a container in detached mode (`-d`), mapping ports (`-p`), and giving it a custom name.
*   `docker ps` — List all running containers.
*   `docker ps -a` — List all containers (running and stopped).
*   `docker stop <container_id_or_name>` — Stop a running container.
*   `docker start <container_id_or_name>` — Start a stopped container.
*   `docker restart <container_id_or_name>` — Restart a container.
*   `docker rm <container_id_or_name>` — Remove a stopped container.
*   `docker rm -f <container_id_or_name>` — Force remove a running container.

### Inspection & Debugging
*   `docker logs -f <container_id_or_name>` — Follow container log output.
*   `docker exec -it <container_id_or_name> <command>` — Run a command inside a running container (e.g., `docker exec -it my_container sh` or `bash`).
*   `docker inspect <object_name_or_id>` — Return low-level information on Docker objects (containers, images, volumes, networks).
*   `docker stats` — Display a live stream of container resource usage statistics.

### Cleanup
*   `docker system prune -a --volumes` — Remove all unused containers, networks, images (both dangling and unused), and optionally volumes. **Use with caution!**

## 3. Dockerfile Syntax & Best Practices

A `Dockerfile` is a text document that contains all the commands a user could call on the command line to assemble an image.

```dockerfile
# 1. Base Image
FROM bun:1.1-alpine AS base

# 2. Set Working Directory
WORKDIR /usr/src/app

# 3. Copy Dependency Manifests
COPY package.json bun.lock ./

# 4. Install Dependencies
RUN bun install --frozen-lockfile

# 5. Copy Application Source Code
COPY . .

# 6. Expose Port
EXPOSE 3000

# 7. Define Startup Command
CMD ["bun", "run", "index.ts"]
```

### Best Practices:
*   **Use `.dockerignore`:** Exclude large folders like `node_modules` or local build artifacts from being copied into the build context.
*   **Minimize Layers:** Combine commands (e.g., `RUN apt-get update && apt-get install -y ...`) to avoid creating unnecessary image layers.
*   **Use Specific Tags:** Instead of `latest`, use specific versions (e.g., `node:20-alpine`) to ensure builds are reproducible.
*   **Use Multi-stage Builds:** Separate the build environment from the final runtime environment to keep the final production image size small.
*   **Run as Non-Root User:** For better security, switch to a non-root user (e.g., `USER node`) in production images.

## 4. Docker Compose Overview

Docker Compose is a tool for defining and running multi-container Docker applications using a YAML file.

```yaml
version: '3.8'

services:
  web:
    build: .
    ports:
      - "3000:3000"
    environment:
      - NODE_ENV=development
      - REDIS_URL=redis://cache:6379
    depends_on:
      - cache

  cache:
    image: "redis:alpine"
    ports:
      - "6379:6379"
```

*   `docker compose up` — Build, (re)create, start, and attach to containers for a service.
*   `docker compose up -d` — Start services in the background.
*   `docker compose down` — Stop and remove containers, networks, images, and volumes created by `up`.
*   `docker compose logs -f` — View logs for all running services in real-time.
